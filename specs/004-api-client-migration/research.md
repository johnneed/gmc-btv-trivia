## Decision 1: Current state of `src/data/trivia-api.ts`

**Current**: Exports `fetchTrivia()` — fetches `VITE_API_URL + "trivia.json"`, returns raw `Quiz[]` with no timeout or auth support.

**Required changes**:
- Rename `fetchTrivia` → `fetchGames` (public games list)
- Add `fetchGame(id: string)` (single published game)
- Add `fetchAllGames(nonce: string)` (all games including drafts, authenticated)
- Replace `VITE_API_URL + "trivia.json"` with `(window.trailTriviaConfig?.apiBase ?? import.meta.env.VITE_API_BASE_URL) + "/games"`
- Add `AbortController` with 10s timeout to all three functions
- Add `X-WP-Nonce` header on `fetchAllGames`
- Throw a typed `UnauthorizedError` on HTTP 401 so the loader slice can distinguish it

**Decision**: Create a custom `UnauthorizedError` class extending `Error`. This is the simplest way to distinguish 401 from 5xx in the thunk's rejected handler without passing structured data through `createAsyncThunk`'s error serialization.

**Alternatives considered**:
- Return `{ error: "unauthorized" }` from the thunk: rejected — breaks the `Quiz[]` return type contract.
- Use `rejectWithValue` in the thunk: viable, but requires changing the thunk signature. The `UnauthorizedError` pattern is simpler and keeps the data layer pure.

## Decision 2: `loader-slice.ts` status union and thunk changes

**Current**: `status: "idle" | "loading" | "failed"`. The `fetchQuizzes.fulfilled` case applies `filterPublished` + `sortByDateDesc` to the raw result from `fetchTrivia()`.

**Required changes**:
- Extend status union: `"idle" | "loading" | "failed" | "unauthorized"`
- `fetchQuizzes.rejected` handler: if `action.error.name === "UnauthorizedError"`, set `status = "unauthorized"`; otherwise `status = "failed"`
- Update thunk to import `fetchGames` (not `fetchTrivia`)
- Remove `filterPublished` from the thunk — the WP REST API public endpoint returns only published games. `sortByDateDesc` is retained (API order is not guaranteed).
- The thunk should NOT catch 401 and return a value — let it reject so the `rejected` extraReducer handles it.

**Alternatives considered**:
- Keep `filterPublished` for safety: rejected — it's dead code once the WP API is in place, and YAGNI applies.

## Decision 3: `ErrorBoundary` component

**Current**: No `ErrorBoundary.tsx` exists in `src/app/`. `react-error-boundary` package is already installed (listed in `dependencies`).

**Decision**: Create `src/app/ErrorBoundary.tsx` using `react-error-boundary`'s `ErrorBoundary` component with a `fallbackRender` prop. The fallback renders "Something went wrong. Try again." and a button that calls `resetErrorBoundary()`. Wrap the `<HashRouter>` in `App.tsx` with this boundary.

**Rationale**: `react-error-boundary` is installed and provides `resetErrorBoundary` for free — no custom class component needed. The retry button resets the boundary, which causes the subtree to re-render and re-triggers the `Loader` component's `useEffect` (since the component remounts).

**Alternatives considered**:
- Class-based `componentDidCatch`: rejected — the package is already installed; writing a class component is unnecessary complexity.

## Decision 4: `VITE_API_URL` → `VITE_API_BASE_URL` rename

**Current**: `.env` has `VITE_API_URL=http://localhost:3000/`. The codebase currently uses `VITE_API_URL`.

**Decision**: Rename to `VITE_API_BASE_URL` pointing at the WP REST API path (e.g., `http://localhost:8888/wp-json/trail-trivia/v1` for a local WP install). Update `.env`, `.env.production.local`, and all source references.

**Note**: `VITE_API_URL` was the old trivia.json base. `VITE_API_BASE_URL` is the new WP REST API base — a fundamentally different path structure. Keeping the old name would be misleading.

## Decision 5: `console.log` stripping in production build

**Current**: `vite.config.ts` has no `define` section. `console.log` calls in source are included in production bundles.

**Decision**: Add `build: { define: { 'console.log': 'void 0' } }` to `vite.config.ts` for the production build only. This replaces `console.log` calls with no-ops at bundle time.

**Rationale**: The MIGRATION_PLAN.md specifies this exact mechanism. It's the idiomatic Vite approach and requires no additional dependency.

**Alternatives considered**:
- ESLint `no-console` rule: enforces it at lint time but doesn't strip from bundles — rejected as insufficient.
- Terser `drop_console`: Vite 5 uses esbuild by default, not Terser — rejected.

## Decision 6: `filterPublished` thunk removal impact

The `filterPublished` transform was added in Phase 2 to filter out future-dated and unpublished quizzes from the raw trivia.json response. The WP REST API public endpoint (`GET /games`) will only return `status: "published"` games with `publishDate <= now` (enforced server-side in Phase 4). Removing client-side filtering is correct and reduces a layer of duplication.

However, Phase 4 is not yet implemented. During Phase 3 development, the developer will be testing against a mock or a dev WordPress instance. The test suite will mock `fetchGames()` to return the expected shape, so `filterPublished` removal doesn't break tests.

**Decision**: Remove `filterPublished` from the thunk now. Add a note in the test suite that the API is assumed to return only published games.
