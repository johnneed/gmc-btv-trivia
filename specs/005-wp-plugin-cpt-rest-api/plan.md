# Implementation Plan: WP Plugin — CPT, Taxonomy & REST API

**Branch**: `005-wp-plugin-cpt-rest-api` | **Date**: 2026-06-26 | **Spec**: [spec.md](spec.md)

**Input**: Feature specification from `specs/005-wp-plugin-cpt-rest-api/spec.md`

## Summary

Fill the four Phase 4 class stubs (`Trail_Trivia_Post_Type`, `Trail_Trivia_REST_API`, `Trail_Trivia_Capabilities`, `Trail_Trivia_Settings`) and wire them to WordPress via `trail-trivia.php`. The result is a fully functional REST data layer verifiable entirely via `curl` and WP-CLI — no browser or admin UI required.

## Technical Context

**Language/Version**: PHP 8.0+ · WordPress 6.4 and 6.5

**Primary Dependencies**: WordPress core only — no Composer packages, no third-party libraries

**Storage**: Custom Post Type `trail_trivia_game` + post meta + `trivia_tag` taxonomy + `trail_trivia_settings` option

**Testing**: WP-CLI smoke tests + `curl` against a running WordPress instance (per MIGRATION_PLAN.md Phase 4 deterministic block)

**Target Platform**: WordPress 6.4–6.5 · PHP 8.0+

**Architecture layer**: `wp-plugin/trail-trivia/` — PHP plugin only; no React changes in this phase

## Constitution Check

*GATE: Must pass before opening the implementation PR.*

- [ ] **II. Layered Architecture (PHP)**: Each `class-*.php` file has exactly one responsibility; `Trail_Trivia_REST_API` delegates persistence to private helpers, not inline `get_post_meta()`/`update_post_meta()` in route handlers.
- [ ] **IV. Accessibility & Errors**: All REST failures return `WP_Error` with `code` + `message`; no `wp_die()` for expected errors; no error swallowed silently.
- [ ] **V. YAGNI**: No code written for Phase 5 or 6 features (admin UI, shortcode); stubs left as stubs.
- [ ] **VI. Separation of Concerns**: `class-rest-api.php` handles routing+validation only; `class-settings.php` handles option storage; `class-capabilities.php` handles capability checks; `class-post-type.php` handles registration only.
- [ ] **IX. Naming (PHP)**: All public functions prefixed `trail_trivia_`; constants prefixed `TRAIL_TRIVIA_`; classes in `PascalCase` with `Trail_Trivia_` prefix; files `class-kebab-case.php`; variables `$snake_case`.
- [ ] **Phase shell tests**: All MIGRATION_PLAN.md Phase 4 deterministic commands exit 0 on WP 6.4 and WP 6.5.

## Implementation Order

### Track A — Registration Layer (no dependencies; do first)

1. Implement `Trail_Trivia_Post_Type::register()` in `class-post-type.php` — `register_post_type('trail_trivia_game', [...])` + `register_taxonomy('trivia_tag', 'trail_trivia_game', [...])`
2. Implement `Trail_Trivia_Capabilities::has_manage_cap()` in `class-capabilities.php` — returns `current_user_can('manage_trail_trivia')`
3. Implement `Trail_Trivia_Settings` in `class-settings.php` — `register_setting()`, `get_settings()`, `get_plugin_info()`, `update_settings($data)`
4. Wire all three to WordPress in `trail-trivia.php` bootstrap (add `add_action('init', ...)` and `add_action('rest_api_init', ...)` calls)
5. Gate: `wp plugin deactivate trail-trivia && wp plugin activate trail-trivia` — exits 0; `php -l` on all PHP files — no errors

### Track B — REST API: Read Endpoints (requires Track A complete)

6. Implement `Trail_Trivia_REST_API::register_routes()` — register all 9 routes with `register_rest_route()`
7. Implement `build_game_response($post)` private helper — maps WP post + meta to Quiz JSON shape
8. Implement `get_games_handler()` — public games list with `date_query`
9. Implement `get_game_handler($request)` — single game by UUID
10. Implement `get_all_games_handler($request)` — all non-trashed games, cap-gated
11. Implement `get_settings_handler($request)` — settings + plugin info
12. Gate: `curl -s http://localhost/wp-json/trail-trivia/v1/games` returns `[]`; `curl` for single game returns 404

### Track C — REST API: Write Endpoints (requires Track B complete)

13. Implement `validate_questions($questions)` private helper — validates array shape, returns `true|WP_Error`
14. Implement `save_game_data($post_id, $data)` private helper — `update_post_meta` for questions, tags (taxonomy sync), schema version
15. Implement `get_game_by_uuid($uuid)` private helper — `WP_Query` by `_trivia_original_id` meta
16. Implement `create_game_handler($request)` — POST /games; generates UUID; saves all data; returns 201
17. Implement `update_game_handler($request)` — PUT /games/{id}; full replacement
18. Implement `patch_game_handler($request)` — PATCH /games/{id}; accepts only status/title/tags
19. Implement `delete_game_handler($request)` — DELETE /games/{id}; `wp_trash_post()`
20. Implement `put_settings_handler($request)` — validates only `gamesPerPage` accepted; HTTP 400 on extra fields
21. Gate: All MIGRATION_PLAN.md Phase 4 deterministic tests exit 0

### Track D — Cleanup (parallel with Track C)

22. Update `uninstall.php` — remove `trail_trivia_settings` option; remove `manage_trail_trivia` cap from all users on uninstall

## Complexity Tracking

No constitution violations. All classes have single responsibility; no third-party dependencies; no cross-layer imports.

**Note on `Trail_Trivia_Capabilities`** (stub marked "Phase 5"): Phase 4 requires capability checks in the REST API. The class stub is implemented minimally now (`has_manage_cap()` only); the grant/revoke UI is deferred to Phase 5. This is additive, not a change to Phase 5 scope.
