## Decision 1: Plugin bootstrap wiring

**Current state**: `trail-trivia.php` includes all class files but does not instantiate any classes or hook them to WordPress actions. The classes exist only as empty stubs.

**Decision**: Add a `Trail_Trivia_Plugin` bootstrap class (or a simple function) in `trail-trivia.php` that:
1. Instantiates `Trail_Trivia_Post_Type` and calls `register()` on `init` (priority 0)
2. Instantiates `Trail_Trivia_REST_API` and calls `register_routes()` on `rest_api_init`
3. Instantiates `Trail_Trivia_Settings` and calls `register()` on `init`

`Trail_Trivia_Capabilities` has no init hook — it is a stateless helper providing `has_manage_cap()`.

**Rationale**: The simplest structure that gets classes wired without a service locator or dependency injection container. Each class registers its own hooks on construction or via a `register()` method.

**Alternatives considered**:
- Single static `init()` method on a bootstrap class: rejected — inline functions in `trail-trivia.php` are simpler and equally maintainable for this plugin's scope.

## Decision 2: Game UUID — `_trivia_original_id` as the canonical `id`

**Current state**: The domain `Quiz` type has `id: string` (UUID). WP post IDs are integers. The React app routes use `quiz.id` (UUID) not the WP post ID.

**Decision**: On every game creation via `POST /games`, generate a UUID v4 using `wp_generate_uuid4()` (available since WP 4.7) and store it in `_trivia_original_id` post meta. All API responses return this UUID as the `id` field. The WP post ID is internal and never exposed.

**Phase 7 compatibility**: When Phase 7 imports `trivia.json`, the existing UUIDs from that file are stored in `_trivia_original_id` — no conflict.

## Decision 3: `publishDate` — WP storage format and filtering

**Current state**: WP stores dates in `post_date` as `Y-m-d H:i:s` (local time). The domain model uses Unix ms.

**Decision**:
- Write: client submits Unix ms integer → plugin converts to WP datetime: `date('Y-m-d H:i:s', intval($ms) / 1000)`
- Read: `strtotime(get_post_date('Y-m-d H:i:s', $post_id)) * 1000` → returned as integer
- WP's `'future'` post status is NOT used. Games submitted with `status: "published"` are always stored with `post_status = 'publish'` regardless of `publishDate`. The `GET /games` public endpoint applies an explicit `date_query` to exclude future-dated published games.

**Rationale**: Using WP's scheduled post mechanism (`'future'` status, `publish_future_post` hook) adds complexity for a feature not needed here. Explicit date filtering in the REST handler is simpler and testable.

## Decision 4: `post_status` ↔ `Quiz.status` mapping

| WP `post_status` | Quiz `status` | API visibility |
|---|---|---|
| `'publish'` | `'published'` | Public + authenticated |
| `'draft'` | `'draft'` | Authenticated only (`manage_trail_trivia`) |
| `'trash'` | — | Not returned by API |

On DELETE: `wp_trash_post($post_id)` → sets `post_status = 'trash'`. Post is NOT permanently deleted.

## Decision 5: Tag sync strategy

**Decision**: On every `create_game` and `update_game`:
1. Call `wp_set_object_terms($post_id, $tags, 'trivia_tag')` to sync taxonomy terms
2. Call `update_post_meta($post_id, '_trivia_tags', wp_json_encode($tags))` to sync denormalized meta
3. Both operations happen in the same request, inside the REST handler

Read path: read from `_trivia_tags` post meta (fast, no taxonomy join query needed). The taxonomy is used by WP for native `tax_query` filtering (Phase 5 admin list).

## Decision 7: Questions validation strategy

**Decision**: Validate `questions` array inline in the REST handler using a private `validate_questions(array $questions): true|WP_Error` method on `Trail_Trivia_REST_API`. Rules:
- Must be a non-empty array of exactly 5 items
- Each item must have: `questionText` (non-empty string), `choices` (array of exactly 4, each with non-empty `text`), `correctAnswerIndex` (int 0–3), `answerText` (non-empty string)
- Each choice must have: `id` (non-empty string) — generated by the factory if absent; `text` (non-empty string)
- Optional question fields: `answerImage` (string or null), `answerImageAlt` (string or null), `answerImageCaption` (string or null)

On validation failure: return `new WP_Error('invalid_questions', descriptive message, ['status' => 400])`.

**Rationale**: WordPress REST API's `validate_callback` parameter is sufficient for simple param types, but deep array validation for nested structures is cleaner as explicit PHP code. No third-party JSON Schema library is needed.

## Decision 8: `Trail_Trivia_Capabilities` scope in Phase 4

The stub is marked "Phase 5" but Phase 4's REST API requires capability checks. The solution: implement only the necessary method in Phase 4 (`has_manage_cap()` → `current_user_can('manage_trail_trivia')`). Phase 5 adds grant/revoke UI methods to the same class.

---

## Decision 9: `Trail_Trivia_Settings` scope in Phase 4

The settings REST endpoints (GET and PUT `/settings`) are listed in Phase 4 deliverables. Therefore `Trail_Trivia_Settings` gets its settings storage logic in Phase 4. The admin page UI is Phase 5.

Settings are stored as a single WordPress option: `trail_trivia_settings` (serialized array).
Default: `['gamesPerPage' => 10]`.

GET returns: `{ gamesPerPage: int, version: string, wpMinimum: string, phpMinimum: string }` (read-only fields come from constants defined in `trail-trivia.php`).

PUT accepts only `gamesPerPage`. Any other key in the body returns HTTP 400.
