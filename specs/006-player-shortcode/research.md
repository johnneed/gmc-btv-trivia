## Decision 1: JavaScript output format — IIFE vs ES module

**Decision**: IIFE (Immediately Invoked Function Expression) format for the player bundle.

**Rationale**: WordPress's `wp_enqueue_script()` loads scripts as `<script src="...">` tags without `type="module"`. ES module output requires `wp_script_add_data($handle, 'type', 'module')` after enqueue. While WP 6.4+ supports `type="module"`, IIFE format is simpler:
- No `wp_script_add_data()` call needed
- No dynamic imports (everything bundled into one file)
- No browser module resolution edge cases
- Rollup disables code-splitting for IIFE, producing a single deterministic `index.js`

**Alternatives considered**:
- ES module format with `wp_script_add_data`: works on WP 6.4+, but adds a PHP call and relies on browser module resolution — rejected for simplicity
- UMD format: designed for library use, not app entrypoints — rejected as wrong tool

## Decision 2: Vite player build configuration

**Decision**: Add a separate `react-app/vite.player.config.ts` that defines the IIFE build targeting `../wp-plugin/trail-trivia/assets/player`. The `build:player` npm script in `package.json` references this config file.

**Key settings**:
- `build.outDir`: `../wp-plugin/trail-trivia/assets/player`
- `build.emptyOutDir`: `true` (clean old files before each build)
- `build.rollupOptions.output.format`: `'iife'`
- `build.rollupOptions.output.entryFileNames`: `'index.js'` (no hash)
- `build.rollupOptions.output.assetFileNames`: `'index[extname]'` (produces `index.css`)
- `esbuild.drop: ['console']`: preserved from main config (strips our console.* calls)
- IIFE `name`: `'TrailTrivia'` (the global name; not used since it's a self-contained app)

**Rationale**: Separate config file rather than a mode/env flag is cleaner — the player build is structurally different (IIFE, different outDir) not just a parameter change.

## Decision 3: Shortcode PHP asset enqueueing strategy

**Decision**: Enqueue assets inside the `render()` shortcode callback directly (not on `wp_head` or `init`). WordPress's enqueue system deduplicates calls automatically, so multiple shortcode instances on the same page do not produce duplicate `<script>` tags.

```php
public function render(): string {
    // Enqueue inside shortcode callback — WordPress deduplicates
    wp_enqueue_script('trail-trivia-player', TRAIL_TRIVIA_PLUGIN_URL . 'assets/player/index.js', [], TRAIL_TRIVIA_VERSION, true);
    wp_add_inline_script('trail-trivia-player', 'window.trailTriviaConfig = ...', 'before');
    wp_enqueue_style('trail-trivia-player', TRAIL_TRIVIA_PLUGIN_URL . 'assets/player/index.css', [], TRAIL_TRIVIA_VERSION);
    return '<div id="trail-trivia-root"></div>';
}
```

This satisfies FR-006 (assets only on shortcode pages) because `wp_enqueue_script` is only called when the shortcode is rendered — never on other pages.

**Alternatives considered**:
- Enqueue on `wp_enqueue_scripts` action with `has_shortcode()` check: more complex, requires the post content to be available at enqueue time (works for posts, fragile for widgets/blocks) — rejected for simplicity.

## Decision 4: `window.trailTriviaConfig` injection

**Decision**: Use `wp_add_inline_script('trail-trivia-player', $config_js, 'before')` to inject the config object immediately before the player script runs. This matches the constitution Architecture section exactly:

```php
wp_add_inline_script(
    'trail-trivia-player',
    'window.trailTriviaConfig = ' . wp_json_encode([
        'apiBase' => rest_url('trail-trivia/v1'),
        'nonce'   => wp_create_nonce('wp_rest'),
    ]) . ';',
    'before'
);
```

`rest_url('trail-trivia/v1')` returns the correct URL for the current WP installation. `wp_create_nonce('wp_rest')` creates a valid REST nonce for the current user (anonymous or logged-in).

## Decision 5: Player script load position (header vs footer)

**Decision**: Load player script in the footer (`$in_footer = true` in `wp_enqueue_script`). This ensures the DOM is available when the script runs, allowing React to mount into `#trail-trivia-root` without needing `DOMContentLoaded` event handling.

---

## Decision 6: `assets/player/` directory creation

**Decision**: Create `wp-plugin/trail-trivia/assets/player/.gitkeep` to establish the directory in version control. The actual `index.js` and `index.css` files are generated by `npm run build:player` and committed alongside the plugin code — they are not gitignored.

**Rationale**: WordPress plugins are deployed as file directories; the built assets are part of the plugin deliverable, not generated at runtime. Committing them is standard WordPress plugin practice.

## Decision 7: `Trail_Trivia_Shortcode` bootstrap wiring

**Decision**: Add shortcode instantiation to the existing `init` hook in `trail-trivia.php`:

```php
add_action('init', function() {
    (new Trail_Trivia_Post_Type())->register();
    (new Trail_Trivia_Settings())->register();
    (new Trail_Trivia_Shortcode())->register();  // Phase 6 addition
});
```

The `register()` method calls `add_shortcode('trail_trivia', [$this, 'render'])`.
