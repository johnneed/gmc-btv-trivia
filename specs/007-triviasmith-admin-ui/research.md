# Research: TriviaSmith Admin UI (Phase 5)

Generated by `/speckit-plan` — 2026-06-26

---

## Decision 1: Admin bundle format — ESM vs IIFE

**Decision**: ES module (ESM) format for the admin bundle, with `wp_script_add_data('trail-trivia-admin', 'type', 'module')` to set `type="module"` on the enqueued script.

**Rationale**: The admin app is large and benefits from code splitting (game list, editor, settings are 3 separate views). IIFE disables Rollup's code splitting, forcing a single large file. ESM gives the browser lazy-loading of routes and is supported natively on WP 6.4+ target. Player uses IIFE because it's a simple single-page app embedded on public pages with no routing needs; admin is a full SPA with multiple routes and complexity that warrants ES module splitting.

**Alternatives considered**: IIFE (same as player) — rejected because code splitting is materially beneficial for an app this size.

## Decision 2: Drag-to-reorder library

**Decision**: Install `@dnd-kit/core` and `@dnd-kit/sortable`. No other drag library is installed.

**Rationale**: `react-beautiful-dnd` is deprecated (Atlassian archived it). `@dnd-kit` is the actively maintained modern replacement, has excellent React integration, supports accessibility (keyboard navigation), and produces small bundles. The spec requires a keyboard-accessible alternative (FR-032); dnd-kit's `useSortable` hook natively supports keyboard activation via the `KeyboardSensor`.

**Alternatives considered**: `react-beautiful-dnd` — deprecated, rejected. Native HTML5 drag API — lacks keyboard accessibility, rejected.

## Decision 3: Admin app directory structure

**Decision**: All admin React source lives in `react-app/src/admin/`. This is a separate entry point from `react-app/src/index.tsx` (the player). The admin app has its own Redux store, router, and feature components.

```
react-app/src/admin/
  index.tsx             ← entry point, mounts to #trail-trivia-admin-root
  App.tsx               ← HashRouter + admin routes
  store/
    index.ts            ← configureStore for admin
    games/              ← games.slice.ts, games.selectors.ts, games.thunks.ts
    editor/             ← editor.slice.ts (form state, autosave status)
    settings/           ← settings.slice.ts
  features/
    game-list/          ← GameList, filter tabs, search, pagination
    game-editor/        ← GameEditor, accordion, publish sidebar, autosave
    settings/           ← SettingsPage (general, access, about panels)
    preview/            ← PreviewModal (mini player tree)
  components/
    question-card/      ← QuestionCard (accordion item)
    image-preview/      ← ImagePreview + Lightbox
    tag-input/          ← TagInput chip component
    publish-sidebar/    ← PublishSidebar metabox
    confirmation-dialog/ ← ConfirmationDialog for trash
```

**Rationale**: Clean separation from the player. The admin app is an independent SPA; intermingling with player `src/features/` would violate single-responsibility.

## Decision 4: Preview Game modal implementation

**Decision**: The preview modal creates a **mini Redux store** pre-populated with the current editor form state (the quiz from `editor.slice.ts`), wraps it in a `<Provider>` + `<MemoryRouter>`, and renders the existing player feature components (`QuizScreen`, `Carousel`, etc.) from `src/features/`. No iframe.

**Why mini store**: The player's Smart components (`QuizScreen`) use `useSelector` and `useParams`. Rather than refactoring them into Dumb components (constitution change), we seed a temporary store with the preview quiz data and mount the player subtree inside it. This isolates the preview from the admin's own store.

**Alternatives considered**: Iframe — rejected (only shows saved data, not live form state). Refactoring player components to Dumb — rejected (large change, out of Phase 5 scope, violates YAGNI).

## Decision 5: Admin PHP wiring

**Decision**: `class-admin-ui.php` handles:
1. `admin_menu` hook: `add_menu_page()` for "Trail Trivia" top-level (requires `manage_trail_trivia`), `add_submenu_page()` for "All Games" (requires `manage_trail_trivia`), `add_submenu_page()` for "Settings" (requires `manage_options`).
2. `admin_enqueue_scripts` hook: enqueue `assets/admin/index.js` (ESM) + `assets/admin/index.css` + `wp_add_inline_script()` for `window.trailTriviaAdminConfig` — but ONLY on Trail Trivia admin pages (check the `$hook_suffix` parameter).
3. Each admin page renders `templates/admin-page.php` which outputs `<div id="trail-trivia-admin-root"></div>`. The React HashRouter handles which view to show based on the hash.

**Rationale**: One PHP template for all three admin pages (game list, editor, settings) — the React router handles routing inside. Simpler PHP, more flexible React.

## Decision 6: `window.trailTriviaAdminConfig`

**Decision**: Inject before the admin bundle:

```php
window.trailTriviaAdminConfig = {
  apiBase: rest_url('trail-trivia/v1'),
  nonce: wp_create_nonce('wp_rest'),
  currentUser: {
    id: get_current_user_id(),
    displayName: wp_get_current_user()->display_name,
    isAdmin: current_user_can('manage_options')
  }
};
```

**Rationale**: The admin needs to know who the current user is for the author display in the editor and to conditionally show the Settings link.

## Decision 7: Admin Vite build config

**Decision**: New `react-app/vite.admin.config.ts`:
- `build.outDir: '../wp-plugin/trail-trivia/assets/admin'`
- `build.rollupOptions.input: 'src/admin/index.tsx'`
- `build.rollupOptions.output.entryFileNames: 'index.js'`
- `build.rollupOptions.output.assetFileNames: 'index[extname]'`
- `build.copyPublicDir: false`
- ESM format (default — no `format: 'iife'`)
- `esbuild.drop: ['console']`

The `build:admin` npm script references this config.

## Decision 8: Autosave implementation

**Decision**: A `useEffect` in `GameEditor` with a `setInterval` firing every 60,000 ms. Each tick dispatches a Redux thunk that calls `PATCH /games/{id}` (or `PUT` for full save) with the current form state. After the save completes, the editor dispatch sets `autosaveStatus = 'saved'` with a timestamp. If no game ID yet (new unsaved game), the autosave calls `POST /games` to create it first and stores the returned ID.

---

## Decision 9: Game list pagination strategy

**Decision**: Server-side pagination via `GET /games/all?page=N&per_page=M`. The REST API (Phase 4) currently returns all games. Phase 5 adds `page` and `per_page` query parameters to `GET /games/all`. The admin game list passes `gamesPerPage` from the Settings as `per_page` and the current page number as `page`.

**Note**: This requires a small addition to `class-rest-api.php` (Phase 4 code) to support pagination parameters on `GET /games/all`. This is a Phase 5 implementation concern, not a new endpoint.
