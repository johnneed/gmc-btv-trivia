## Decision 1: Layer violations in `loader-api.ts`

**Decision**: Move `src/features/loader/loader-api.ts` → `src/data/trivia-api.ts` and strip it of all transform logic.

**Rationale**: `loader-api.ts` currently violates three rules simultaneously:
- It lives in `src/features/` (not `src/data/`) — side-effect isolation violation.
- It contains `R.compose` filter/sort logic that belongs in `src/domain/transforms/`.
- It has a `console.log` that must be removed from production code.
The fix is a straight move + split: fetch-only code goes to `src/data/trivia-api.ts`, transform logic goes to `quiz.transforms.ts`.

**Alternatives considered**:
- Rename in-place (`loader-api.ts` → `src/data/trivia-api.ts` via import rewrite only): rejected — the file is structurally wrong in its current form regardless of location.

## Decision 2: Fate of `src/models/` directory

**Decision**: Delete `src/models/` after migrating all imports to `src/domain/`.

**Rationale**: Phase 0 created the canonical types in `src/domain/types/`. The `src/models/` directory is the old CRA structure. It duplicates types with minor differences (e.g., `trivia.type.ts` exists only in models; `app-user.type.ts` and `plugin-settings.type.ts` exist only in domain). All four active consumers (`loader-api.ts`, `loader-slice.ts`, `choice-button.tsx`, `carousel.tsx`, `quiz/index.tsx`) import from `../../models/types` — these all need updating to `../../domain/types`.

**Alternatives considered**:
- Keep both: rejected — active duplication; Phase 3 would import from the wrong layer.

## Decision 3: `extractQuestionTags` in `loader-slice.ts`

**Decision**: Remove `extractQuestionTags` and the `questionTags` / `selectedQuestionTags` state fields from the loader slice.

**Rationale**: The domain `Question` type has no `tags` field (per the finalized schema in MIGRATION_PLAN.md Section 1.1). The `loader-slice.ts` code references `q.tags` on `Question`, which is a type error after Phase 0. No UI currently uses `questionTags` or `selectedQuestionTags`. Removing dead code rather than patching a bug against a type that was intentionally simplified.

**Alternatives considered**:
- Add `tags?: string[]` back to Question type: rejected — the constitution and MIGRATION_PLAN.md explicitly state "no per-question tags" and "Question has no tags field".

## Decision 4: Handling `src/libs/`

**Decision**: Keep `src/libs/` as-is. Move `scrollTop()` call sites to component event handlers (not into `src/data/`); no structural relocation needed.

**Rationale**: The constitution says "Network calls and all I/O MUST live in `src/data/`". `window.scrollTo()` is a DOM side effect, not network I/O. The three helper files (`date-helpers`, `string-helpers`, `window-helpers`) are lightweight pure or DOM utilities used by components and features. The target architecture in Appendix B doesn't mention `src/libs/` but doesn't forbid it either. Relocating them adds churn with no functional benefit in Phase 2; Phase 3's API client migration is the right time to reconsider.

**Alternatives considered**:
- Move to `src/domain/` (pure helpers) and `src/data/` (scroll): rejected — `scrollTop` is not I/O in the constitution sense; adding it to the data layer is overcategorization.

## Decision 5: ARIA — Progress bar and question heading

**Decision**: Add a new `ProgressBar` dumb component in `src/components/progress-bar/`. Add a semantic `<h2>` for question text in `QuestionComponent` inside `carousel.tsx`.

**Rationale**:
- FR-004 requires `role="progressbar"` with ARIA attributes. No progress indicator currently exists in the UI.
- FR-003 requires `aria-labelledby` pointing at the question heading. The question text is currently in a `<div>` — screen readers can't reference a `<div>` reliably as a group label without a proper heading. Adding `<h2>` (or wrapping in a visually-positioned heading) gives the `aria-labelledby` a reliable anchor.

**Alternatives considered**:
- Use `id` on the div and reference it via `aria-labelledby`: technically valid but a `<div>` with an `id` isn't announced as a heading by all screen readers — a semantic `<h2>` is cleaner and accessible by default.

## Decision 6: framer-motion reduced-motion strategy

**Decision**: Use framer-motion's `useReducedMotion()` hook at the point of each animated `motion.div`; when it returns true, pass `{ animate: false }` or zero-duration variants.

**Rationale**: framer-motion 11 ships `useReducedMotion()` which reads `prefers-reduced-motion` from the OS and re-renders on change. This is the idiomatic approach — no custom CSS `@media` query needed. The hook is already available in the installed version.

**Alternatives considered**:
- CSS `@media (prefers-reduced-motion: reduce) { .motion { transition: none } }`: works for CSS transitions but has no effect on JavaScript-driven framer-motion animations — rejected.

## Decision 8: axe-core CLI installation

**Decision**: Install `@axe-core/cli` as a dev dependency (`npm install -D @axe-core/cli`).

**Rationale**: The Phase 2 deterministic test block in MIGRATION_PLAN.md calls `npx axe-core-cli`. The package name is `@axe-core/cli` (not `axe-core-cli`). Adding it as a dev dependency makes the exact binary available without `npx` download on each run and pins the version.

---

## Decision 9: `document.title` and focus management on route change

**Decision**: Add a `useDocumentTitle` effect in each Smart feature component that sets `document.title` on mount. Add a `<main id="main-content">` landmark and use `focus()` on it after route transition via a router listener.

**Rationale**: React Router 6 doesn't update `document.title` automatically. The cleanest approach is a `useEffect` in each feature (Home, QuizList, Quiz, Score) that sets the appropriate title. Focus management requires a stable `<main>` landmark that can be targeted after navigation.
